<h2 id="dannye-prilozheniya">Данные приложения</h2><br/>
<p>И сразу в бой, начнём с хранилища. Единственный тип необходимый для этого приложения — Todo:</p><br/>
<pre><code>import { EventEmitter } from 'cellx';
import { observable } from 'cellx-decorators';

export default class Todo extends EventEmitter {
    @observable text = void 0;
    @observable done = void 0;

    constructor(text, done = false) {
        super();

        this.text = text;
        this.done = done;
    }
}</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
<p>Тут всё предельно просто, парочка наблюдаемых полей, одно содержит текст задачи, другое статус её выполнения.</p><br/>
<p>Наследование от <code>cellx.EventEmitter</code> необходимо на случай если в дальнейшем понадобится подписаться на изменения какого-то поля:</p><br/>
<pre><code>todo.on('change:text', () =&gt; {/* ... */});</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
<p>В данном приложении такого нет и наследование можно убрать, я просто зачем-то всегда пишу его заранее.</p><br/>
<p>Теперь напишем корневое хранилище:</p><br/>
<pre><code>import { EventEmitter, cellx } from 'cellx';
import { observable, computed } from 'cellx-decorators';
import Todo from './types/Todo';

class Store extends EventEmitter {
    @observable todos = cellx.list([
        new Todo('Primum', true),
        new Todo('Secundo'),
        new Todo('Tertium')
    ]);

    @computed doneTodos = function() {
        return this.todos.filter(todo =&gt; todo.done);
    };
}

export default new Store();</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
<p>Здесь уже поинтереснее. Используется <a href="https://github.com/Riim/cellx/blob/master/README.ru.md#cellxobservablelist">cellx.list</a> (алиас для <code>new cellx.ObservableList</code>) — наблюдаемый список, наследует от <code>cellx.EventEmitter</code> и при любом своём изменении генерирует событие <code>change</code>. Наблюдаемое поле получая в качестве значения что-то наследующее от <code>cellx.EventEmitter</code> подписывается на его <code>change</code> и тоже изменяется при этом событии. Всё это значит, что не обязательно использовать встроенные коллекции, можно сделать свои унаследовав их от <code>cellx.EventEmitter</code>. Из коробки есть <code>cellx.list</code> и <a href="https://github.com/Riim/cellx/blob/master/README.ru.md#cellxobservablemap">cellx.map</a>. Отдельным модулем есть индексируемые версии обоих коллекций: <a href="https://github.com/Riim/cellx-indexed-collections">cellx-indexed-collections</a>.</p><br/>
<p>Ещё один новенький — декоратор <code>computed</code>, вычисляемые поля — это самая суть cellx-a — вы просто пишите формулу вычисляемого поля, вам не нужно самому подписываться на <code>done</code> каждого todo при его добавлении и отписываться от него же при удалении, всё это делает cellx пока вы не видите, вам остаётся расслабиться и получать удовольствие описывая самую суть. При этом описание происходит, можно сказать, в декларативном виде — уже не нужно думать о событиях и о том как изменения будут распространяться по системе, всё пишется так, как будто отработает лишь раз. Кроме того cellx очень умный и автоматически делает некоторые хитрые оптимизации: <a href="https://github.com/Riim/cellx/blob/master/README.ru.md#Динамическая-актуализация-зависимостей">динамическая актуализация зависимостей</a> и <a href="https://github.com/Riim/cellx/blob/master/README.ru.md#Схлопывание-и-отбрасывание-событий">схлопывание и отбрасывание событий</a> не допустят избыточных расчётов и обновлений интерфейса. Если делать всё это вручную, код получается довольно объёмным, но, что намного хуже — глючным. Отладкой же cellx-а заниматься приходиться раз в сто лет, он просто работает.</p><br/>
<h2 id="predstavlenie-prilozheniya">Представление приложения</h2><br/>
<p>Переходим к слою отображения. Сначала компонент задачи:</p><br/>
<pre><code>import { observer } from 'cellx-react';
import React from 'react';
import toggleTodo from '../../actions/toggleTodo';
import removeTodo from '../../actions/removeTodo';

@observer
export default class TodoView extends React.Component {
    render() {
        let todo = this.props.todo;

        return (&lt;li&gt;
            &lt;input type="checkbox" checked={ todo.done } onChange={ this.onCbDoneChange.bind(this) } /&gt;
            &lt;span&gt;{ todo.text }&lt;/span&gt;
            &lt;button onClick={ this.onBtnRemoveClick.bind(this) }&gt;remove&lt;/button&gt;
        &lt;/li&gt;);
    }

    onCbDoneChange() {
        toggleTodo(this.props.todo);
    }

    onBtnRemoveClick() {
        removeTodo(this.props.todo);
    }
}</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
<p>Здесь из новенького — декоратор <code>observer</code> из модуля <a href="https://github.com/Riim/cellx-react">cellx-react</a>. Грубо говоря, он просто делает метод <code>render</code> вычисляемой ячейкой и вызывает <a href="https://facebook.github.io/react/docs/component-api.html#forceupdate">React.Component#forceUpdate</a> при её изменении.</p><br/>
<p>Остаётся корневой компонент приложения:</p><br/>
<pre><code>import { computed } from 'cellx-decorators';
import { observer } from 'cellx-react';
import React from 'react';
import store from '../../store';
import addTodo from '../../actions/addTodo';
import TodoView from '../TodoView';

@observer
export default class TodoApp extends React.Component {
    @computed nextNumber = function() {
        return store.todos.length + 1;
    };

    @computed leftCount = function() {
        return store.todos.length - store.doneTodos.length;
    };

    render() {
        return (&lt;div&gt;
            &lt;form onSubmit={ this.onNewTodoFormSubmit.bind(this) }&gt;
                &lt;input ref={ input =&gt; this.newTodoInput = input } /&gt;
                &lt;button type="submit"&gt;Add #{ this.nextNumber }&lt;/button&gt;
            &lt;/form&gt;
            &lt;div&gt;
                All: { store.todos.length },
                Done: { store.doneTodos.length },
                Left: { this.leftCount }
            &lt;/div&gt;
            &lt;ul&gt;{
                store.todos.map(todo =&gt; &lt;TodoView key={ todo.text } todo={ todo } /&gt;)
            }&lt;/ul&gt;
        &lt;/div&gt;);
    }

    onNewTodoFormSubmit(evt) {
        evt.preventDefault();

        let newTodoInput = this.newTodoInput;

        addTodo(newTodoInput.value);

        newTodoInput.value = '';
        newTodoInput.focus();
    }
}</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
<p>Здесь ещё парочка вычисляемых полей, отличаются от <code>Store#doneTodos</code> они лишь тем, что поля из которых они вычисляются лежат не на текущем экземпляре (<code>this</code>), а где-то в другом месте, cellx никак не ограничивает в этом плане, эти поля можно спокойно переместить в <code>Store</code> и всё так же будет работать. Определять, где должно лежать поле лучше по его сути — если поле специфично для какого-то определённого компонента, то пусть в нём и вычисляется, светиться в общем хранилище ему нет смысла. В данном случае я бы <code>#leftCount</code> перенёс в хранилище, оно вполне может пригодиться где-то ещё, а <code>#nextNumber</code> вполне неплохо смотриться и здесь.</p><br/>
<h2 id="biznes-logika-prilozheniya">Бизнес-логика приложения</h2><br/>
<p>В экшенах cellx никак не используется, поэтому я максимально их упростил, получился даже не Flux, а какой-то MVC в терминах Flux-а. Надеюсь вы мне простите это упрощение.</p><br/>
<h2 id="rezultat">Результат</h2><br/>
<p>В данном случае приложение совсем простое и написать его так же просто можно и без cellx-а (никаких подписок на каждый <code>done</code> здесь не потребуется), при дальнейшем же усложнении связей в приложении сложность их описания на cellx-e растёт линейно, без него — обычно нет и в какой-то момент приходим к мешанине событий в которой без поллитра не разобраться. Для решения проблемы, кроме реактивного программирования, есть и другие подходы со своими плюсами и минусами, но их сравнение — уже другая история (если кратко, как минимум они проигрывают из-за большого количества лишних вычислений и, как результат, более низкой производительности). </p><br/>
<p>В общем-то по коду это всё, ещё раз ссылка на результат: <a href="https://riim.github.io/react-cellx-todo-app/">жмяк</a> (<a href="https://github.com/Riim/react-cellx-todo-app">код</a>).</p><br/>
<h2 id="sravnenie-s-drugimi-bibliotekami">Сравнение с другими библиотеками</h2><br/>
<h3 id="mobx">MobX</h3><br/>
<p>Чаще всего спрашивают отличия от <a href="https://github.com/mobxjs/mobx">MobX</a>. Это наиболее близкий аналог и отличий немного: </p><br/>
<ol>
<li>cellx примерно в 10 раз <a href="https://github.com/Riim/cellx/blob/master/README.ru.md#Тест-производительности">быстрее</a>. </li>
<li>В статье про <a href="http://habrahabr.ru/post/235121/">атомы</a> я подсмотрел методы/опции put и pull, позволяющие ячейкам уметь чуть больше: <a href="https://github.com/Riim/cellx/blob/master/README.ru.md#Синхронизация-значения-с-синхронным-хранилищем">синхронизация значения с синхронным хранилищем</a>, <a href="https://github.com/Riim/cellx/blob/master/README.ru.md#Синхронизация-значения-с-асинхронным-хранилищем">синхронизация значения с асинхронным хранилищем</a>, <a href="https://github.com/Riim/cellx/issues/7#issuecomment-245661130">про pull</a>. У MobX я ничего похожего не нашёл. </li>
<li>MobX лучше интегрирован с React-ом, в отличии от cellx-а он так же как-то вклинивается в слой бизнес-логики приложения. Я так и не понял зачем он там, но видимо зачем-то нужен.</li>
</ol><br/>
<h3 id="kefirjs-baconjs">Kefir.js, Bacon.js</h3><br/>
<p>Тут отличия более существенны. Отставание в скорости ещё больше, но важнее не это. Эти библиотеки предлагают создавать вычисляемые ячейки несколько иначе, в, наверное, более функциональном виде. То, что на cellx-e будет выглядеть так:</p><br/>
<pre><code>var val2 = cellx(() =&gt; val() + 1);</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
<p>На этих библиотеках превратиться в что-то вроде (псевдокод, как там точно я не помню, да и не суть):</p><br/>
<pre><code>var val2 = val.lift(add(1));</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
<p>Плюс в более красивом, человекочитаемом коде, минус в заметно большем пороге входа, так как теперь нужно запомнить 100500 методов на все случаи жизни (можно конечно обходиться и каким-то минимальным набором). </p><br/>
<p>В тоже время в cellx-е есть возможность добавить ячейкам свои методы и ничто не мешает довести его до уровня этих библиотек, можно сказать, что он более низкоуровневый.</p><br/>
<h2 id="podval">Подвал</h2><br/>

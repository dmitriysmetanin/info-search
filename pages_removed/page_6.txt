article-formatted-body_version-1"><div xmlns="http://www.w3.org/1999/xhtml">Иногда можно услышать такие разговоры: никаких принципиальных изменений в Java 8 не произошло и лямбды это старые добрые анонимные классы щедро посыпанные синтаксическим сахаром. Как бы не так! Предлагаю сегодня поговорить, в чём отличие лямбд от анонимных классов. И почему попасть себе в ногу стало всё-таки сложнее.<br/>
<a name="habracut"></a><br/>
Чтобы не отнимать время у тех, кто считает что уже освоился с анонимными функциями, простенькая задачка. Чем отличаются два фрагмента кода ниже:<br/>
<br/>
<pre><code class="java">public class AnonymousClass {
    public Runnable getRunnable() {
        return new Runnable() {
            @Override
            public void run() {
                System.out.println("I am a Runnable!");
            }
        };
    }

    public static void main(String[] args) {
        new AnonymousClass().getRunnable().run();
    }
}
</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
И второй фрагмент:<br/>
<br/>
<pre><code class="java">public class Lambda {
    public Runnable getRunnable() {
        return () -&gt; System.out.println("I am a Runnable!");
    }

    public static void main(String[] args) {
        new Lambda().getRunnable().run();
    }
}
</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
Если можете сходу ответить — решайте сами, хотите ли читать дальше.<br/>

<h3>Анализируем</h3><br/>
Что-нибудь бросилось в глаза? Та-та-та-дам…<br/>
<br/>

<br/>
<pre><code>5: invokespecial #3          // Method AnonymousClass$1."&lt;init&gt;":(LAnonymousClass;)V

<i>Лямбда:</i><br/>
<br/>
<pre><code>0: invokedynamic #2,  0      // InvokeDynamic #0:run:()Ljava/lang/Runnable;</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
Кажется анонимный класс захватил при создании <a href="http://docs.oracle.com/javase/specs/jvms/se8/html/jvms-4.html#jvms-4.3.2">ссылку</a> на порождающий его экземпляр:<br/>
<br/>
<pre><code>AnonymousClass$1."&lt;init&gt;":(LAnonymousClass;)V</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
и будет держать её, пока всесильный Сборщик Мусора™ не пометит его как недостижимый и не освободит от этого бремени. Хотя никак эта ссылка внутри не используется, но вот такой он анонимный жадина.<br/>
<br/>
А если серьёзно, то здесь потенциальная утечка памяти, если вы отдаёте экземпляр анонимного класса во внешний мир. С лямбдами это произойдёт только в том случае, если вы явно или неявно ссылаетесь на <i>this</i> в теле анонимной функции. В противном случае, как в этом примере, <b>лямбда ссылки на вызывающий её экземпляр не держит</b>.<br/>
<br/>
<b>Делаем своими руками.</b> Предлагаю всем читателям провести эксперимент и посмотреть что будет в каждом из случаев, если к строке добавить вызов <i>.toString()</i> у порождающего экземляра.<br/>
<br/>
<h3>Как в ногу-то попасть? Обещал рассказать!</h3><br/>
Самый простой способ напороться на потенциальную утечку памяти — это использовать внутри лямбды нестатические методы внешнего класса, если вам в реальности неинтересно его внутреннее состояние:<br/>
<br/>
<pre><code class="java">public class LambdaCallsNonStatic {
    public Runnable getRunnable() {
        return () -&gt; {
            nonStaticMethod();
        };
    }

    public void nonStaticMethod() {
        System.out.println("I am a Runnable!");
    }

    public static void main(String[] args) {
        new LambdaCallsNonStatic().getRunnable().run();
    }
}
</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
Лямбда получит ссылку на экземпляр класса её вызывающий (хотя будет создана один раз, но об этом ниже):<br/>
<br/>
<pre><code>1: invokedynamic #2,  0      // InvokeDynamic #0:run:(LLambdaCallsNonStatic;)...
</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
<div class="spoiler"><b class="spoiler_title">Декомпиляция LambdaCallsNonStatic.class</b><div class="spoiler_text"><pre><code>Compiled from "LambdaCallsNonStatic.java"

<b>Решение:</b> объявить используемый метод статическим или вынести его в отдельный утильный класс.<br/>
<br/>
<h3>И всё?</h3><br/>
Нет, есть ещё одна замечательная плюшка у лямбд по сравнению с анонимными классами. Если вы когда-нибудь работали в застенках кроваво-энтерпрайзной конторы и не дай боже́ писали такое:<br/>
<br/>
<pre><code class="java">Collections.sort(list, new Comparator&lt;Integer&gt;() {
            @Override
            public int compare(Integer o1, Integer o2) {
                return -Integer.compare(o1, o2);
            }
        });
</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
То подходил к вам о мудрейший тимлид и говорил:<br/>
<br/>
<i>Не экономно ты, <s>Фёдор</s> &lt;имя разработчика&gt;, ресурсы корпоративные расходуешь. Давай мы это зарефакторим по-взрослому.</i><br/>
<br/>
Ведь новый экземпляр компаратора будет создаваться каждый раз при работе этого фрагмента кода. В результате получалась такая портянка:<br/>
<br/>
<pre><code class="java">public class CorporateComparators {
    public static Comparator&lt;Integer&gt; integerReverseComparator() {
        return IntegerReverseComparator.INSTANCE;
    }

    private enum IntegerReverseComparator implements Comparator&lt;Integer&gt; {
        INSTANCE;

        @Override
        public int compare(Integer o1, Integer o2) {
            return -Integer.compare(o1, o2);
        }
    }
}

...

Collections.sort(list, CorporateComparators.integerReverseComparator());</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
Удобнее же стало, всё в своём файлике теперь лежит и переиспользовать можно. С последним соглашусь, но удобнее стало разве что если у вас DDR4 вместо серого вещества в голове. Читабельность такого кода не просто падает, а летит в тартарары со сверхзвуковой.<br/>
<br/>
С лямбдами можно держать логику ближе к месту непосредственного использования и не платить за это сверху:<br/>
<br/>
<pre><code class="java">Collections.sort(list, (i1, i2) -&gt; -Integer.compare(i1, i2));</code><div class="code-explainer"><a href="https://sourcecraft.dev/" class="tm-button code-explainer__link" style="visibility: hidden;"></a></div></pre><br/>
Анонимная функция, не захватывающая значений из внешнего контекста, будет лёгкой и создаваться один раз. Хотя спецификация не обязывает конкретную реализацию виртуальной машины к такому поведению (<a href="http://docs.oracle.com/javase/specs/jls/se8/html/jls-15.html#jls-15.27.4">15.27.4. Run-Time Evaluation of Lambda Expressions</a>), но в Java HotSpot VM наблюдается именно это.<br/>
